# Learning-curve experiment

Quantifies how F1 scales with paired-game training data so we can
answer: **"how many more games do we need to reach the target F1?"**

## TL;DR — data status right now

- 22 video files in `data/videos/`
- 31 GT CSVs in `data/ground_truth/`
- **8 paired** (video + matching GT): `1922466, 2078805, 2092329, 2132246, 2137794, 2137820, 2141538, 2146953`
- **24 orphan GTs** that just need their video fetched via `hudl-fetch` — these are the cheap path to more pairs

A meaningful learning curve needs ≥4 well-spaced training-size points
that span ~5× in size. With 8 pairs (hold out 3 → train pool 5) the
curve points are crammed at [1, 3, 4, 5] — extrapolation is unreliable.
**Get to ~16 paired games first** (hold out 4 → train pool 12 → curve
[3, 6, 9, 12]). Best path: run `hudl-fetch`'s `grab_videos_batch.py`
against the 24 orphan match_ids.

## Files

| | |
|---|---|
| `splits.py`   | generate `splits.json` (test holdout + train pool + curve sizes) |
| `eval.py`     | strict-window F1 evaluator — reuses `eval/eval_cv_seg_output.py` primitives, no duplication |
| `run_curve.py`| `record` / `plot` / `show` subcommands |
| `splits.json` | generated; controls the experiment |
| `results.json`| generated; accumulates (train_size, F1) points |
| `curve.png`   | generated by `run_curve.py plot` |

## Workflow

1. **Generate splits** (once per data-pool change):

   ```bash
   python3 training/learning_curve/splits.py
   ```

   Stratifies by GT shot count so test isn't skewed toward low-shot games.

2. **Train + predict for each curve size**, externally. For curve size `N`:
   - Take the first `N` match_ids from `splits.json["train_pool"]`
   - Train your chosen model on those games' (video + GT) pairs
   - Run inference on `splits.json["test_match_ids"]`
   - Save predictions as a CSV — required columns:

     ```csv
     vID,start_s,end_s
     2078805,123.4,135.7
     2078805,210.0,219.0
     2092329,55.5,68.2
     …
     ```

   Optional columns: `confidence`, `team`. `vID` must match the
   `match_id` (i.e., the numeric stem of `gt_<id>.csv`).

3. **Record the point:**

   ```bash
   python3 training/learning_curve/run_curve.py record \
       --train-size 6 \
       --predictions runs/yolo_temporal_n6/preds.csv \
       --notes "yolo+temporal head, 30 epochs, hard-neg-mining"
   ```

   Eval is deterministic and uses the SAME IoU primitives as the
   project's documented F1 (so the number is directly comparable to
   the 0.422 baseline).

4. **Plot the curve:**

   ```bash
   python3 training/learning_curve/run_curve.py plot \
       --baseline-f1 0.422 \
       --target-f1   0.90
   ```

   Fits a power-law (`F1 = a − b·N^−c`) and projects how many games
   you'd need to hit `--target-f1`. If the target exceeds the
   asymptote `a`, the script tells you — that means **more data won't
   save you, change the architecture**.

5. **Show table whenever:**

   ```bash
   python3 training/learning_curve/run_curve.py show
   ```

## What I'd run first

Once you're at ≥16 pairs:

- **Architecture candidate A** — per-frame YOLO (HockeyAI fine-tune)
  + hard-negative mining + per-video calibration. Documented ceiling
  per `specs/EVAL_NOTES.md` ≈ 0.70–0.85.
- **Architecture candidate B** — short-clip temporal model
  (TimeSformer or 3D-CNN over 2-3 s clips centred on shot moment).
  Realistic ceiling ≈ 0.80–0.92. This is the only path with the
  asymptote above 0.90.

Run candidate A first — cheaper to set up. If its curve asymptote
fits below 0.90, that's the empirical signal to switch to B rather
than fetching more games.

## Re-running splits when data grows

`splits.py` is deterministic given `--seed`. Don't change the seed
between runs without good reason — that would scramble the test
holdout and invalidate all previously-recorded curve points.

When you add more pairs:
- If you want to KEEP existing curve points comparable, leave
  `test_match_ids` as-is and just expand the train pool / extend
  `curve_sizes`.
- If you want a clean rerun, delete `results.json` and re-record.
